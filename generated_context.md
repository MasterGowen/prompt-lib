# Глубокое код-ревью

## Навык
Проведение детального анализа кода с выявлением проблем архитектуры, безопасности, производительности и читаемости.

## Процесс ревью

### 1. Первичный анализ
- Понимание контекста: что делает этот код, какая его цель
- Оценка структуры: логичность разбиения на функции/классы
- Проверка Naming: понятность имен переменных, функций, классов

### 2. Детальная проверка
- **Безопасность**: уязвимости, инъекции, обработка чувствительных данных
- **Производительность**: сложные операции в циклах, лишние аллокации, N+1 проблемы
- **Читаемость**: сложность строк, вложенность, дублирование
- **Тестируемость**: можно ли покрыть тестами, есть ли зависимости
- **Совместимость**: backward compatibility, миграции

### 3. Формат обратной связи
Для каждой найденной проблемы:
```
📍 Файл: строка X-Y
⚠️ Проблема: краткое описание
📝 Почему это плохо: объяснение последствий
💡 Решение: конкретный пример исправления
🎯 Приоритет: high/medium/low
```

### 4. Итоговое резюме
- Список критических проблем (требуют исправления обязательно)
- Рекомендации по улучшению (желательно, но не блокирует)
- Положительные моменты (что сделано хорошо)

## Пример запроса
"Проведи код-ревью этого pull request с фокусом на безопасность и производительность"

## Примечания
- Не придираться к стилю, если он консистентен в проекте
- Предлагать альтернативы, а только не критику
- Учитывать контекст проекта (startup vs enterprise)

# Валидация входных данных (Python)

## Сниппет промпта
Используй этот паттерн для генерации кода валидации:

```
При работе с пользовательским вводом всегда применяй следующий паттерн:
1. Явно объяви ожидаемую схему данных (тип, диапазон, формат)
2. Используй early validation — отклоняй невалидные данные как можно раньше
3. Возвращай понятные ошибки с указанием поля и причины
4. Для сложных структур используй библиотеки валидации (pydantic, marshmallow)
5. Логируй попытки невалидного ввода для мониторинга атак

Пример на Python:
```python
from pydantic import BaseModel, EmailStr, Field, validator
from typing import Optional

class UserInput(BaseModel):
    email: EmailStr
    age: int = Field(ge=0, le=150)
    username: str = Field(min_length=3, max_length=20)
    
    @validator('username')
    def username_alphanumeric(cls, v):
        if not v.isalnum():
            raise ValueError('Username must be alphanumeric')
        return v

# Использование
try:
    user = UserInput(**request_data)
except ValidationError as e:
    return {"error": "Invalid input", "details": e.errors()}
```
```

## Когда использовать
- Генерация API endpoints
- Обработка форм
- Парсинг конфигов
- Работа с внешними данными

# Сборка контекста для LLM

## Цель
Автоматизированная подготовка релевантного контекста из базы знаний перед отправкой задачи нейросети.

## Воркфлоу

### Шаг 1: Определение задачи
Пользователь формулирует задачу. AI определяет:
- Какие типы промптов нужны (roles, skills, workflows)
- Какие теги релевантны
- Требуется ли конкретный эксперимент или сравнение

### Шаг 2: Фильтрация
Запуск скрипта сборки с параметрами:
```bash
# Пример: нужны верифицированные навыки по Python
python utils/build_context.py --status verified --tags python --type skill

# Пример: нужен конкретный эксперимент
python utils/build_context.py --experiment EXP-001

# Пример: всё для код-ревью
python utils/build_context.py --tags code-review --status verified
```

### Шаг 3: Валидация контекста
Проверка сгенерированного файла:
- Размер не превышает лимиты модели
- Контекст релевантен задаче
- Отсутствуют противоречивые инструкции

### Шаг 4: Формирование промпта
Структура итогового сообщения:
```
1. Системная роль (из roles/)
2. Навыки (из skills/)
3. Специфичный контекст задачи
4. Ограничения и требования
5. Формат ожидаемого ответа
```

### Шаг 5: Отправка и итерация
- Отправка промпта модели
- Анализ ответа
- При необходимости — уточнение с добавлением контекста

## Автоматизация в Roo Code
Создать алиас в настройках Roo:
```json
{
  "name": "build-context",
  "command": "python utils/build_context.py --status verified --output generated_context.md"
}
```

## Метрики успеха
- Время подготовки контекста < 30 секунд
- Релевантность ответа > 90%
- Минимум итераций уточнения

## Примечания
- Всегда фильтровать по `status: verified` для продакшена
- Для экспериментов использовать отдельный вывод
- Не включать deprecated промпты без явной необходимости
